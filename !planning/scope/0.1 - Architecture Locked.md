# Version 0.1 - Architecture Locked

## Cil

Verze **0.1** je interni MVP urcene k overeni cilove architektury aplikace a spravnosti celeho datoveho toku. Neni to zmensena verejna verze produktu. Je to technicky milnik, ktery musi dokazat, ze aplikace umi spolehlive zpracovat realna financni data od importu az po portfolio a dashboard.

V teto fazi uz nestaci "smerovat k Python backendu". Soucasti verze **0.1** je i dokonceni hlavni backendove hranice:

- frontend zustava v `Next.js` a `TypeScript`,
- hlavni backendova logika bezi v `Pythonu`,
- `Next.js` neobsahuje domenovou business logiku jako source of truth,
- import, ledger, snapshots, portfolio a dashboard workflow obsluhuje `Python API`.

Aplikace v teto verzi neni urcena pro verejne pouziti.

---

## Proc tato verze existuje

Smyslem verze **0.1** je potvrdit, ze dalsi vyvoj bude stavet na spravne architekture, stabilnim datovem modelu a overenych backendovych hranicich. Tato verze ma odstranit nejvetsi technicka rizika co nejdrive:

- jestli je Python backend schopny obslouzit hlavni use-cases,
- jestli importy produkuji konzistentni data,
- jestli holdings, cash a snapshoty davaji spravne vysledky,
- jestli portfolio a dashboard cti data ze stabilniho read modelu,
- jestli lze aplikaci dal rozsirovat bez dalsiho drzeni kriticke logiky v Next.js route handlerech.

---

## Hlavni cile

- Overit cilovou architekturu aplikace.
- Dokoncit hlavni backendovy prechod pro MVP do `Python API`.
- Vytvorit stabilni backend rozdeleny po domenach.
- Overit datovy model a hranice mezi importy, ledgerem, holdings, snapshoty a read modely.
- Implementovat kompletni tok dat od importu az po portfolio a dashboard.
- Nastavit zakladni testovaci infrastrukturu pro parsery, vypocty a backend API.
- Pripravit projekt na dalsi rozvoj bez nutnosti drzet business logiku v `Next.js` backendu.

---

## Architektura pro 0.1

Verze **0.1** potvrzuje tento cilovy tvar:

- `Next.js` slouzi pro UI, routing, formulare a klientskou praci s daty.
- `Python API` je hlavni aplikacni backend.
- `PostgreSQL` je centralni persistence vrstva.
- `Rust` neni povinnou soucasti implementace 0.1, ale rozhrani backendu a vypoctu musi s jeho pozdejsim pouzitim pocitat.

Pro verzi **0.1** plati:

- frontend cte produkcni data z `Python API`,
- portfolio, dashboard, import a snapshot workflow bezi pres `Python API`,
- `Next.js` nesmi byt source of truth pro domenovou logiku,
- pokud nejake `Next.js API` route zustanou, mohou fungovat jen jako tenke technicke adaptery:
  - forward/proxy,
  - session passthrough,
  - transportni kompatibilita bez vlastnich business rules.

---

## Hlavni MVP scenar

Hlavni overovany scenar verze **0.1** je tento:

1. Uzivatel se prihlasi do aplikace.
2. Vytvori ucet s hlavni menou.
3. Nahraje podporovany export.
4. Frontend odesle data do `Python API`.
5. Python backend provede parsing, validaci a vytvori standardizovane udalosti.
6. Python backend ulozi data, prepocita holdings a cash, vytvori nebo obnovi snapshoty a pripravi read modely.
7. Frontend zobrazi portfolio a dashboard z dat vracenych `Python API`.
8. Uzivatel vidi spravna data bez zavislosti na puvodni `Next.js` business logice.

Tento scenar musi fungovat na ciste databazi i na opakovanem lokalnim testovani.

## Povinne technicke minimum

### Backend

- Samostatne bezici `Python API`.
- Modulove rozdeleni backendu minimalne pro:
  - accounts,
  - imports,
  - ledger / events / movements,
  - holdings,
  - snapshots,
  - portfolio read model,
  - dashboard read model,
  - prices / FX.
- Jasne API kontrakty mezi frontendem a Python backendem.
- Odstraneni kriticke domenove logiky z `Next.js` route handleru.

### API minimum

Python backend musi mit alespon endpointy nebo ekvivalentni use-cases pro:

- autentizaci nebo session bridge,
- accounts,
- import batch a import status,
- portfolio read model,
- dashboard read model,
- snapshot rebuild nebo refresh.

### Frontend

- `Next.js` frontend funguje nad `Python API`.
- Frontend nesmi pro hlavni MVP scenar zaviset na stare `TypeScript` business logice.
- UI muze byt utilitarni, ale musi korektne reprezentovat data a stavy.

### Vyvoj a provozni minimum

- Docker nebo ekvivalentni lokalni spusteni.
- Konfiguracni system pro lokalni backend provoz.
- Automaticke testy.
- Linting a formatovaci standard.
- Zakladni dokumentace pro lokalni spusteni a backend hranici.

## Datova pravidla a invariants

Aby bylo mozne hovorit o "spravnych datech", musi verze **0.1** dodrzet tato pravidla:

- Kazdy ucet ma hlavni menu.
- Account-level portfolio a dashboard zobrazuji hlavni hodnoty v hlavni mene uctu.
- Snapshoty nesmi byt natvrdo vedene v `CZK`.
- `investovano` a `vlozeno` se pocita kurzem ke dni udalosti, ne dnesnim kurzem.
- Dnesni nebo live portfolio hodnota vychazi z posledniho denniho snapshotu a z udalosti po tomto snapshotu.
- Cash a hodnoty se musi zachovat i po jednotlivych menach, ne jen v jedne zobrazovaci mene.
- Historicky graf portfolia ma cist data ze snapshotu nebo z read modelu nad snapshoty, ne prepocitavat celou historii pri kazdem cteni.
- Importni parsery musi vracet jednotny datovy model.
- Nepodporovane radky nesmi zmizet; musi byt evidovane jako parse issues.
- Opakovany import nesmi nekontrolovane zdvojovat data.
- Multi-file import jedne historie musi post-processing holdings a snapshotu provadet jako jednu davku, ne po kazdem souboru zvlast.

## Kriteria dokonceni

Verze **0.1** je dokoncena ve chvili, kdy plati vsechny tyto body:

- Uzivatel se muze prihlasit a pracovat v `Next.js` frontend aplikaci.
- Dokaze vytvorit financni ucet s hlavni menou.
- Dokaze nahrat podporovany export.
- Frontend odesila hlavni MVP workflow do `Python API`.
- Python backend spravne zpracuje import a vytvori standardizovana data.
- Python backend ulozi udalosti, prepocita holdings a cash a vytvori snapshoty.
- Portfolio stranka funguje nad `Python API`.
- Dashboard funguje nad `Python API`.
- Snapshot creation nebo rebuild bezi mimo `Next.js` business logiku.
- Puvodni `Next.js` backend neni nutny pro hlavni MVP scenar.
- Portfolio a dashboard vraceji overitelne spravna data nad znamymi fixture a realnymi testovacimi importy.
- Hlavni scenare jsou pokryty automatickymi testy.
- Architektura umoznuje pridani noveho parseru nebo modulu bez zasadnich zmen existujiciho systemu.
