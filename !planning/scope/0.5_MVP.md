# Version 0.5 - Production MVP

## Cil

Verze **0.5** je prvni produkcne pouzitelne MVP. Nejde o plne hotovy produkt, ale o verzi, kterou lze rozumne nasadit pro realne pouzivani s jasne definovanym rozsahem, spolehlivym hlavnim workflow a dostatecnou provozni jistotou.

---

## Proc tato verze existuje

Po `0.4` je aplikace pripraven a na private beta provoz. Ve verzi **0.5** se musi potvrdit:

- ze hlavni use-cases jsou dostatecne spolehlive pro produkcni nasazeni,
- ze uzivatel muze aplikaci pouzivat bez pomoci vyvojare,
- ze tym umi aplikaci provozovat, diagnostikovat a bezpecne rozsirovat,
- ze scope MVP je jasne ohraniceny a realisticky.

---

## Hlavni cile

- Dokoncit produkcni MVP scope.
- Stabilizovat hlavni uzivatelske workflow od uctu po portfolio.
- Zajistit duveryhodna data, predvidatelne chovani a provozni dohled.
- Omezit zname tridy chyb na prijatelnou uroven.
- Jasne oddelit, co MVP umi spolehlive a co zustava mimo rozsah.

---

## Scope 0.5

### Core product workflow

- Uzivatel se prihlasi, vytvori ucet, nahraje data a dostane portfolio a dashboard bez technickych obchazek.
- Hlavni importni zdroje jsou stabilni a podporovane.
- Portfolio, dashboard a account prehled se chovaji konzistentne a predvidatelne.

### Data trust

- Snapshoty, holdings, cash, invested a deposited jsou povazovany za produkcne duveryhodne v podporovanem scope.
- Hlavni cisla se nemeni kvuli znamym internim nekonzistencim.
- Uzivatel vidi, kdy jsou data neuplna, fallbackovana nebo ve zpracovani.

### Product usability

- Hlavni obrazovky jsou citelne, rychle a dostatecne vysvetluji stav dat.
- Import feedback, parse issues a chybove stavy jsou srozumitelne.
- Zakladni account management a bezne operace jsou ergonomicke.

### Operations

- Python backend je hlavni produkcni backend pro MVP.
- Nasazeni, logovani, monitoring a zakladni incident workflow existuji.
- Kriticke joby jsou dohledatelne a umi bezpecny retry nebo rebuild.

---

## Povinne technicke minimum

- Stabilni `Python API` pro account, import, portfolio, dashboard a snapshot workflow.
- Frontend plne funguje nad oficialnimi backend kontrakty.
- Zakladni monitoring, error tracking a provozni alerting pro kriticke chyby.
- Dokumentovane nasazeni, rollback a troubleshooting.
- Test coverage pro core flows dostatecna k beznemu releasu.

---

## Produkcni pravidla pro MVP

- Scope MVP musi byt uzivatelum i tymu jasne komunikovan.
- Nepodporovane scenare musi selhavat predvidatelne, ne potichu.
- Kriticke hodnoty v portfoliu a dashboardu musi byt reprodukovatelne.
- Bezpecnostni a account boundary pravidla nesmi byt porusena kvuli pohodli implementace.
- Rozvoj po 0.5 musi jit delat bez navratu business logiky do `Next.js` backendu.

---

## Testovani a validace

Minimalni scope testovani:

- end-to-end testy hlavnich uzivatelskych workflow,
- regresni testy parseru a snapshot logiky,
- API integracni testy pro klicove produkcni kontrakty,
- smoke testy produkcniho nasazeni,
- provozni testy pro retry, rebuild a failure recovery hlavniho import workflow.

Minimalni validacni scenare:

- novy podporovany uzivatel projde hlavnim workflow bez asistence vyvojare,
- import vrati spravne a srozumitelne vysledky,
- portfolio a dashboard jsou konzistentni i po vice importech a refreshich,
- produkcni problem jde dohledat, popsat a resit bez improvizace.

## Co neni soucasti 0.5

- rozsahle enterprise features
- pokrocile planovani, budgety a forecast moduly
- plna automaticka synchronizace vsech bank a brokeru
- high-scale distribuovana architektura

---

## Kriteria dokonceni

Verze **0.5** je dokoncena ve chvili, kdy:

- aplikace je schopna produkcniho MVP provozu v jasne definovanem scope,
- hlavni workflow funguje stabilne a bez znamych kritickych datovych rozporu,
- frontend, Python backend a databaze tvori konzistentni a provozne udrzitelny celek,
- tym umi system nasadit, provozovat, diagnostikovat a bezpecne dale rozvijet.
