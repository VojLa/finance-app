# Version 0.3 - Usable Internal Product

## Cil

Verze **0.3** meni aplikaci z technicky spravneho systemu na pouzitelny interni produkt. Architektura a cisla uz musi byt dostatecne stabilni, aby se dalsi prace soustredila na ergonomii, workflow a bezny uzivatelsky provoz.

Tato verze nema za cil verejny release. Ma zajistit, ze aplikaci lze realne pouzivat pri kazdodenni praci bez potreby technickych obchazek.

---

## Proc tato verze existuje

I pres spravna data muze byt aplikace stale neprijemna na pouzivani. Ve verzi **0.3** se musi potvrdit:

- ze import workflow je srozumitelny a dohledatelny,
- ze uzivatel rozumi stavu dat a chybam,
- ze portfolio a dashboard jsou prakticky pouzitelne,
- ze bezna prace nevyzaduje rucni kontrolu databaze nebo logu.

---

## Hlavni cile

- Udelat z hlavniho workflow pouzitelny interni produkt.
- Zlepsit UX importu, portfolia a dashboardu.
- Vyrazne rozsirit realnou importni pouzitelnost aplikace.
- Dodat zakladni account management a beznou provozni ergonomii.
- Zobrazit stavy systemu tak, aby uzivatel chapal, co se deje a proc.
- Snizit potrebu rucnich technickych zasahu pri kazdodennim pouziti.

---

## Scope 0.3

### Import workflow

- Import je asynchronni, se srozumitelnym stavem zpracovani.
- Uzivatel vidi batch status, vysledek, parse issues a chyby.
- Multi-file import je pouzitelny v UI jako jeden logicky krok.
- Rebuild nebo refresh snapshotu je dostupny jako interni nastroj.

### Rozsireni parseru a importni pokryti

- Aplikace podporuje vyrazne sirsi sadu instituci, ne jen puvodni tri zdroje.
- Cilem verze `0.3` neni maximalizovat pocet parseru, ale pokryt vsechny hlavni instituce cilove skupiny.
- Parsery se nepridavaji nahodne; prednost maji instituce, ktere realne odemknou pouzitelnost aplikace pro kazdodenni praci.
- Kazdy novy parser musi pouzivat stejny importni kontrakt, parse issues a post-processing pipeline jako existujici zdroje.
- Pridani noveho parseru ma byt natolik predvidatelne, aby trvalo maximalne nekolik hodin, ne dny nebo refaktor cele importni vrstvy.

### Custom CSV pro nepodporovane instituce

- Uzivatel musi umet nahrat vlastni `custom CSV` pro instituci, kterou aplikace nativne nepodporuje.
- Aplikace musi mit jasne definovany custom import format jako fallback vrstvu.
- Musi existovat srozumitelny navod, jak tento `custom CSV` vytvorit:
  - povinne sloupce,
  - podporovane typy radku,
  - format data, meny, castky a identifikatoru,
  - priklady validnich radku.
- Custom CSV import nesmi byt jen technicka pomucka pro vyvoj; musi byt navrzen jako realne pouzitelny workflow pro uzivatele.

### Navody pro podporovane instituce

- Pro kazdou oficialne podporovanou instituci musi existovat navod, jak spravny CSV export ziskat.
- Navod musi popsat:
  - kde v dane sluzbe export najit,
  - jaky format nebo filtr vybrat,
  - kterym exportum se vyhnout, pokud jsou pro parser nepodporovane,
  - pripadne zname odchylky mezi verzemi exportu.
- Cilem je, aby uzivatel dokazal bez technicke pomoci vytvorit spravny export jak pro podporovanou instituci, tak pro custom CSV fallback.

### Portfolio a dashboard UX

- Portfolio stranka zobrazuje hodnotu, cash podle men, invested, deposited a historicky vyvoj bez technickych zkratek.
- Dashboard a portfolio pouzivaji jednotny jazyk a jednotna cisla.
- Grafy jsou dostatecne rychle a citelne pro beznou interni praci.
- Chybejici nebo fallback data jsou rozeznatelna od potvrzenych dat.

### Ucty a bezne operace

- Uzivatel umi vytvorit, upravit a archivovat ucet.
- Zakladni prehled uctu je srozumitelny a stabilni.
- Hlavni mena uctu je viditelna a respektovana v celych read modelech.

### Diagnostika a feedback

- Uzivatel vidi chybove a uspesne stavy bez nutnosti cist backend logy.
- Failnute importy a snapshot operace jsou dohledatelne.
- Aplikace ma zakladni internI support workflow pro re-run nebo kontrolu stavu.

---

## Povinne technicke minimum

- Frontend je plne napojen na `Python API` pro hlavni interni use-cases.
- TanStack Query nebo ekvivalentni data-fetching vrstva ma jasne definovane loading, error a refresh stavy.
- Import, portfolio a dashboard endpointy jsou dostatecne stabilni pro bezne opakovane pouziti.
- Dokumentace lokalniho pouziti a beznych provoznich kroku je citelna.
- Existuje dokumentace pro:
  - podporovane instituce a zpusob exportu jejich CSV,
  - custom CSV format pro nepodporovane instituce,
  - pridani noveho parseru bez poruseni existujici importni architektury.

---

## UX pravidla pro 0.3

- Uzivatel musi rozumet, kdy jsou data aktualni, ve zpracovani nebo neuplna.
- Chybove stavy musi byt popsane srozumitelne, ne jen technickym stack tracem.
- Dulezite hodnoty se zobrazuji v hlavni mene uctu, ale menovy rozpad zustava dohledatelny.
- Import nesmi pusobit jako jednorazovy technicky script, ale jako stabilni workflow.
- Importni vrstva musi byt prakticky pouzitelna i pro uzivatele mimo tri puvodni zdroje.
- Pokud instituce neni primo podporovana, uzivatel musi pochopit, ze ma k dispozici custom CSV fallback a jak ho spravne pripravit.

## Testovani a validace

Minimalni scope testovani:

- integracni testy hlavniho import workflow,
- fixture testy pro novou vlnu parseru a jejich edge cases,
- validacni testy custom CSV formatu,
- testy UI stavu pro loading, success a error scenare,
- endpoint testy pro account, import status, portfolio a dashboard,
- smoke testy hlavniho interniho user flow.

Minimalni validacni scenare:

- novy uzivatel vytvori ucet a nahraje podporovany export bez technicke asistence,
- uzivatel nepodporovane instituce dokaze podle navodu pripravit a nahrat `custom CSV`,
- dokumentace podporovanych instituci staci k tomu, aby uzivatel nasel a vyexportoval spravny CSV soubor,
- po importu vi, co bylo nacteno, co bylo preskoceno a co selhalo,
- portfolio a dashboard jsou citelne a odpovidaji importovanym datum,
- bezna oprava nebo re-run nevynucuje rucni zasah do DB.

## Kriteria dokonceni

Verze **0.3** je dokoncena ve chvili, kdy:

- hlavni workflow aplikace je pouzitelne bez technickych obchazek,
- aplikace ma dostatecne siroke importni pokryti pro realne pouziti, vcetne dalsich `20` az `30` parseru nebo ekvivalentne siroke prakticke podpory,
- aplikace pokryva vsechny hlavni instituce cilove skupiny nebo pro ne ma robustni `custom CSV` fallback,
- pridani noveho parseru je rychle, predvidatelne a nevyzaduje zasah do zakladni architektury,
- custom CSV fallback je definovany, zdokumentovany a pouzitelny,
- podporovane instituce maji srozumitelne navody pro export spravneho CSV,
- import, portfolio a dashboard davaji uzivateli srozumitelnou zpetnou vazbu,
- bezne interni pouziti nevyzaduje cteni DB nebo logu pro kazdy krok,
- architektura zustava konzistentni a frontend nevytahuje business logiku zpet do Next.js.
