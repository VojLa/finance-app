# 5F-B2 – Anycoin grouped investment normalization

## Metadata

- Milestone: `0.1 - Architecture Locked`
- Parent: `5F - transaction classification`
- Dependency: merged Step 5F-B1, PR #31, commit `3e5b70d`
- Size: L; keep implementation focused on Anycoin grouping only
- Source of truth: Python `imports` module

## Goal

Replace unsafe per-row Anycoin normalization with a deterministic batch-level grouping stage. Group `trade payment`, `trade fill`, and optional `trade refund` rows by `Order ID`, produce one canonical schema-v2 investment event for each complete order, preserve standalone crypto deposits/withdrawals, and keep posting/ledger work out of scope.

## Why this is batch-level

An Anycoin trade is not represented by one CSV row. The legacy behavior groups rows by order ID and computes net asset and cash movements. The current Python normalization service calls `normalize_import_row` independently for each row, which cannot safely construct an Anycoin trade. Step 5F-B2 therefore adds a source-specific batch normalizer invoked inside the existing normalization transaction.

## Raw Anycoin row contract

Parse exact aliases using trim, Unicode case-folding, and whitespace collapse for header matching:

- type: `Type`, `Operation`, `Transaction type`
- order ID: `Order ID`, `Order id`, `OrderId`, `Order`
- date: `Date`, `Time`, `Created at`
- amount: `Amount`, `Quantity`
- currency/asset: `Currency`, `Asset`
- external ID: `anycoin TX ID`, `Transaction ID`, `TX ID`

Each parsed row must retain only bounded canonical fields in memory. Raw source data stays on `ImportRow.rawData`.

All amounts use finite `Decimal`. Never use `float` or tolerance/epsilon comparisons.

Currencies and crypto symbols are uppercase. Fiat detection must use an explicit shared allowlist, not symbol length or guesswork. Include at least currencies already supported by the application and fixtures; unknown identifiers are treated as assets unless explicit evidence says otherwise.

## Exact row-type allowlist

After type normalization:

### Grouped trade rows

- `trade payment` → payments bucket
- `trade fill` → fills bucket
- `trade refund` → refunds bucket

Every grouped row requires a non-empty order ID.

### Standalone rows

- `deposit`
- `withdrawal`

Standalone events require date, non-zero amount, and currency. They represent crypto asset movement only when the currency is non-fiat. Fiat deposit/withdrawal rows must return review because cash semantics are not yet established for Anycoin exports.

### Neutral temporary rows

- `payment block`
- `payment block refund`
- `withdrawal_block`
- `withdrawal_unblock`

These rows are deterministic neutral rows. They do not create posting intents or deduplication candidates.

### Unsupported rows

All other types return structured `unsupported_anycoin_row` review outcomes. Messages must not echo source type, order ID, values, or identifiers.

## Grouping invariants

Group only rows from the current locked import batch. Do not group across batches.

A complete trade requires:

- one non-empty order ID shared by all members;
- at least one payment row;
- at least one fill row;
- valid dates and finite non-zero amounts on contributing rows;
- exactly one non-fiat asset currency after aggregation;
- exactly one fiat cash currency after aggregation;
- non-zero net asset amount;
- non-zero net cash amount.

Multiple rows in the same bucket are allowed and must be summed exactly with `Decimal`.

Refund rows participate in net cash/asset calculations according to their currency. They must not be ignored merely because their type is `trade refund`.

Reject groups with:

- missing order ID;
- missing payment or fill bucket;
- invalid or missing required field;
- multiple net asset currencies;
- multiple net fiat currencies;
- zero net asset or zero net cash after aggregation;
- contradictory direction between asset and cash nets;
- duplicate/conflicting external IDs when a deterministic canonical ID cannot be selected.

## Direction and canonical event

Let:

- `asset_net` be the exact sum of all rows in the single asset currency;
- `cash_net` be the exact sum of all rows in the single fiat currency.

A valid trade must have opposite signs:

- `asset_net > 0` and `cash_net < 0` → `buy`;
- `asset_net < 0` and `cash_net > 0` → `sell`.

Any other sign combination requires review.

Canonical magnitudes:

- quantity = `abs(asset_net)`;
- total amount = `abs(cash_net)`;
- price amount = total / quantity using `Decimal`;
- asset symbol = asset currency;
- asset type hint = `crypto` based on explicit Anycoin source semantics;
- total/price currency = fiat currency;
- date = latest valid fill date; if unavailable, latest valid member date;
- order ID is preserved in the canonical payload;
- external ID selection is deterministic.

External ID precedence:

1. unique non-empty external ID from an asset fill row;
2. unique non-empty external ID from any fill row;
3. unique non-empty external ID from payment/refund rows;
4. otherwise order ID.

If a precedence level contains multiple different IDs, do not pick by row order; return review with `conflicting_external_id`.

Use the common schema-v2 investment shape introduced by 5F-B1. Add `order_id` as an optional canonical field if not already present in the common intent.

Example:

  "kind": "investment_event",
  "date": "2026-07-23T10:00:00+00:00",

## Anchor and member row persistence

No new table or enum is introduced.

For every complete grouped trade:

1. Choose the anchor deterministically:
   - lowest `row_number` among fill rows;
   - tie-break by stable row ID.
2. Persist the canonical schema-v2 event and deduplication key on the anchor.
3. Keep the anchor status `pending`.
4. Mark every other member row `skipped`.
5. Member rows have no deduplication key.
6. Store a compact bounded schema-v2 group-member marker in `normalized_data`, containing only:
   - `schema_version`;
   - `source`;
   - `kind = "group_member"`;
   - `order_id`;
   - `anchor_row_id`;
   - normalized member role (`payment`, `fill`, `refund`).
7. Member markers must not be classifiable into posting intents.

Do not use `duplicate` for group members; duplicate is reserved for Step 5E cross-event duplicate detection.

## Fully refunded groups

A group with payments and refunds, no fills, and exact zero net amount in every involved currency is fully refunded.

For fully refunded groups:

- create no event;
- mark all member rows `skipped`;
- store bounded `kind = "fully_refunded_group"` markers with order ID;
- produce no deduplication key;
- do not count the group as review or parser failure.

No epsilon comparison is allowed; exact `Decimal` net must equal zero.

## Incomplete or invalid groups

For any non-neutral group that cannot safely become a canonical event:

- mark every member row `needs_review`;
- clear normalized event data and deduplication keys;
- persist structured, safe validation errors;
- use group-level codes such as:
  - `missing_order_id`;
  - `incomplete_order`;
  - `multiple_asset_currencies`;
  - `multiple_fiat_currencies`;
  - `zero_group_net`;
  - `contradictory_trade_direction`;
  - `conflicting_external_id`;
  - `invalid_anycoin_row`.

Do not include group summaries, raw values, amounts, currencies, IDs, or row contents in error messages.

## Standalone deposit and withdrawal

A valid standalone non-fiat row produces one schema-v2 event on the same row:

- positive amount + `deposit` → `asset_transfer` with incoming semantics represented by positive quantity;
- negative amount + `withdrawal` or positive provider magnitude according to documented source convention → canonical outgoing `asset_transfer`.

Do not infer direction solely from sign without reconciling the exact action. The canonical action/intent must preserve enough information for Step 5G to create the correct movement direction. If the current shared `InvestmentAction.asset_transfer` cannot express direction, add a bounded explicit field such as `asset_direction: in|out` to schema v2 and the posting intent. Apply it consistently to Trading212 transfers as well only if needed for contract correctness, with non-regression tests.

Standalone fiat rows require review with `unsupported_anycoin_fiat_transfer`.

## Classification

Extend `classify_import_row` so:

- valid Anycoin schema-v2 investment events produce immutable complete `InvestmentEventPostingIntent`;
- source must match `anycoin`;
- grouped event invariants are defensively revalidated;
- group-member and fully-refunded markers return structured non-postable review/ignored results and never investment intents;
- schema-v1 Anycoin rows continue returning `investment_normalization_required`;
- Trading212 schema-v2 behavior remains unchanged.

Prefer extracting source-neutral schema-v2 validation rather than duplicating the entire Trading212 classifier. Provider-specific invariants may remain separate.

## Deduplication identity

Deduplication remains account/source scoped.

Grouped trade identity:

- preferred: account + source + order ID;
- if order ID is unavailable, the group is invalid and cannot become a trade;
- external IDs may be retained for traceability but order ID is the stable grouped-event identity.

Standalone identity:

- account + source + external ID when available;
- fallback: date + action + asset symbol + quantity + direction.

Equivalent decimal spellings and raw row order must generate the same key. Reordering rows inside one order must not change the anchor event payload, deduplication key, or computed values, except anchor selection remains based on persisted row number by design.

## Normalization service integration

Preserve the existing authenticated `/normalize` endpoint and transaction boundary.

Inside `ImportNormalizationService.normalize_batch`:

- retain authorization, account/source advisory lock, `FOR UPDATE`, state checks, rollback, and idempotence protections;
- dispatch Anycoin batches to one batch-level grouping function;
- keep per-row normalizer behavior for other sources;
- compute counters from final row outcomes.

Counter semantics:

- `rows_total` = all rows;
- `rows_normalized` = pending canonical event rows only;
- `rows_needs_review` = all rows marked needs_review;
- `rows_failed` = parser-failed rows;
- `rows_skipped` batch counter continues current compatibility semantics and must include parser failures, review rows, neutral rows, group members, and fully refunded rows if that is how current response/batch counters are defined; document the exact behavior and test it.

## Security and determinism

- Treat raw data as untrusted.
- Bound all optional strings.
- Never log raw rows, order IDs, external IDs, asset balances, or amounts.
- No dynamic SQL or filesystem paths from provider data.
- No dependence on dictionary iteration order or CSV input ordering for aggregation.
- Sort groups and member processing deterministically.
- Use exact `Decimal` arithmetic only.
