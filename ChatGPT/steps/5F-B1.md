## Canonical normalized contract

Bank/manual rows keep normalized schema version `1`.

Trading212 rows use normalized schema version `2` with the following stable shape. Exact Pydantic class names may differ, but the serialized JSON contract and semantics must remain equivalent.

```json
{
  "schema_version": 2,
  "source": "trading212",
  "kind": "investment_event",
  "date": "2026-07-23T10:00:00+00:00",
  "action": "buy",
  "external_id": "transaction-id-or-null",
  "raw_action": "Market buy",
  "asset": {
    "symbol": "VWCE",
    "isin": "IE00BK5BQT80",
    "name": "Vanguard FTSE All-World",
    "asset_type_hint": null
  },
  "quantity": "2",
  "price": {"amount": "100.5", "currency": "EUR"},
  "total": {"amount": "201", "currency": "EUR"},
  "fee": null,
  "conversion": null,
  "realized_pnl": null,
  "is_promotional": false,
  "note": null
}
```

Rules:

- Omit or serialize optional sections consistently according to one documented convention; do not mix missing keys and `null` unpredictably.
- All decimal values are strings generated from finite `Decimal` values.
- Quantity, price, total, fee, and conversion-leg amounts are canonical positive magnitudes. Direction comes from `action`.
- Realized P/L remains signed.
- Currency and symbol identifiers are uppercase.
- Do not infer ETF/stock type from an ISIN prefix. `asset_type_hint` is optional and may only contain explicit source evidence.
- Raw input remains in `ImportRow.rawData`; do not duplicate the full raw row in normalized data.

## Trading212 field mapping

Use exact source-field aliases, independently for each semantic field.

### Identity and action

- action: `Action`
- date: `Time`, `Time (UTC)`
- symbol: `Ticker`
- ISIN: `ISIN`
- name: `Name`, `Merchant`, `Merchant name`, `Merchant Name`
- external ID: `ID`, `Transaction ID`
- note: `Notes`, `Note`, `Category`, `Merchant category`, `Card category`, `Card Category`

### Financial fields

- quantity: `No. of shares`, `Quantity`
- price amount: `Price / share`, `Price per share`, `Price`
- price currency: `Currency (Price / share)`, `Currency (Price)`, `Price currency`
- total amount: `Total`, `Amount`
- total currency: `Currency (Total)`, `Currency (Amount)`, `Currency`
- exchange rate: `Exchange rate`, `Exchange Rate`
- conversion from amount: `Currency conversion from amount`
- conversion from currency: `Currency (Currency conversion from amount)`
- conversion to amount: `Currency conversion to amount`
- conversion to currency: `Currency (Currency conversion to amount)`
- realized P/L: `Result`, `Realized P/L`
- realized P/L currency: `Currency (Result)`, `Currency (Realized P/L)`

Fee amount is the sum of the absolute finite decimal values in supported Trading212 fee columns. Fee currency must be resolved deterministically from the corresponding populated fee column; conflicting non-empty fee currencies require review.

Supported fee columns initially include:

- `Currency conversion fee`
- `Stamp duty reserve tax`
- `French transaction tax`
- `Finra fee`

## Action mapping

Use Unicode case-folding, trim, and whitespace collapse followed by exact allowlist lookup. Do not use substring or description inference.

### Trade

Map to canonical `buy`:

- `buy`
- `purchase`
- `market buy`
- `limit buy`
- `stop buy`
- `crypto purchase`

Map to canonical `sell`:

- `sell`
- `sale`
- `market sell`
- `limit sell`
- `stop sell`
- `stop loss`
- `take profit`
- `crypto sale`

### Other supported events

- dividend variants → `dividend`
- interest variants and `spending cashback` → `interest`
- currency conversion / FX conversion / exchange / convert / swap → `currency_conversion`
- explicit asset/internal/portfolio transfer → `asset_transfer`
- fee variants → `fee`
- staking variants → `staking_reward`
- airdrop variants → `airdrop`
- ordinary cash deposit/top-up/funding → `cash_deposit`
- ordinary cash withdrawal → `cash_withdrawal`

### Free-share behavior

The exact free-share/free-stock/referral/promo allowlist may map to `airdrop` only when asset identity and a positive quantity exist. Set `is_promotional = true` and do not create a fee section from promotional-row fee columns.

### Linked cash transaction behavior

The following actions remain `needs_review` with `unsupported_linked_cash_transaction`:

- `card debit`
- `card cost`
- `new card cost`

They must not silently become `cash_withdrawal` or an investment event because their future posting requires a linked ordinary transaction contract.

Unknown actions return a structured unsupported-action review result.

## Required invariants by action

### Buy and sell

Require:

- valid date;
- symbol or ISIN;
- positive quantity;
- positive total amount and total currency;
- optional price must contain both positive amount and currency;
- fee must contain both amount and currency when present.

### Dividend

Require asset identity and positive total amount/currency.

### Cash deposit, cash withdrawal, interest, and fee

Require positive total amount/currency. Asset identity is optional.

### Asset transfer, staking reward, and airdrop

Require asset identity and positive quantity. Total/price may be absent.

### Currency conversion

Require positive `from` and `to` amounts, both currencies, and different currencies. Exchange rate is optional but must be positive when present.

### Realized P/L

When present, amount and currency are both required. Amount may be positive, negative, or zero.

Incomplete pairs, conflicting currencies, invalid decimals, non-finite values, zero required magnitudes, and missing required fields produce field-specific structured review errors. Error messages must not echo raw financial values or identifiers.

## Posting intent

Update the internal immutable `InvestmentEventPostingIntent` so it can carry the complete canonical investment payload needed by Step 5G. Do not reduce an investment event to one ambiguous `amount` and `currency` pair.

The intent must include at least:

- source;
- date;
- canonical event type;
- canonical buy/sell action when relevant;
- asset identity;
- quantity;
- price;
- total;
- fee;
- conversion legs;
- realized P/L;
- external ID;
- promotional flag.

`model_dump(mode="json")` must contain only JSON-safe strings, booleans, nulls, lists, and objects.

## Deduplication identity

Trading212 deduplication remains account- and source-scoped.

- If a non-blank external ID exists, canonical identity is account + source + external ID.
- Otherwise use a stable canonical identity containing date, action, asset identity, quantity, total, conversion legs, and relevant currencies.
- Do not include description, note, name, raw action spelling, or raw field order in fallback identity.
- Equivalent decimal spellings and input column order must generate the same SHA-256 key.
- Financially different rows must not collapse to the same key.

## Existing normalization workflow

The existing authenticated `POST .../{batch_id}/normalize` action remains the only persistence entry point in this step.

- Trading212 valid rows persist normalized schema version `2` and remain `pending` for Step 5E deduplication.
- Invalid or unsupported Trading212 rows persist no normalized payload/key and become `needs_review` with structured validation errors.
- Parser-failed rows remain `failed`.
- Counters and rollback behavior remain compatible with Step 5D.
- Normalization and deduplication continue using the existing account/source PostgreSQL advisory lock.

## Anycoin compatibility
