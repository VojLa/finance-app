# 5G – Canonical posting

## Metadata

- Milestone: `0.1 - Architecture Locked`
- Dependency: Step 5F-C, merged as `16daefcb58314865fb6eedabd4d7b3eb660a56fc`
- Source of truth: Python `imports` workflow and canonical SQLAlchemy domain models
- Size: XL before splitting

## Goal

Consume validated, persisted posting intents from deduplicated import rows and create
canonical database history exactly once.

A successful posting workflow must:

- create either one `Transaction` or one `InvestmentEvent` with its required
  `InvestmentMovement` rows;
- preserve the canonical normalized payload and posting intent for audit;
- mark the source `ImportRow` as `imported` and store the created entity ID;
- finalize deterministic batch counters and status;
- remain idempotent under retries and concurrent requests;
- roll back every entity, row and batch mutation on failure.

## Why 5G is split

The original step combines three different risk domains:

1. transaction persistence and signed-money mapping;
2. investment asset/listing resolution and multi-movement accounting;
3. authenticated batch orchestration, finalization, concurrency and retry behavior.

Implementing all three in one PR would make review and rollback reasoning too broad.
Step 5G is therefore split into:

- **5G-A – canonical transaction posting foundation**
  - validate one persisted transaction intent;
  - re-derive and compare the deterministic classifier result;
  - create one canonical `Transaction`;
  - mark one locked import row as imported;
  - support exact idempotent replay;
  - no public endpoint and no batch finalization.
- **5G-B – canonical investment posting foundation**
  - validate investment intents;
  - resolve or create canonical `Asset` and `AssetListing` records conservatively;
  - create one `InvestmentEvent` and ordered `InvestmentMovement` rows;
  - mark one locked import row as imported;
  - support exact idempotent replay;
  - no public endpoint and no batch finalization.
- **5G-C – batch posting workflow**
  - add `POST /api/v1/accounts/{account_id}/imports/{batch_id}/post`;
  - authorize owner/admin/editor and reject viewer;
  - use the existing account/source advisory lock, batch lock and deterministic row locks;
  - invoke 5G-A and 5G-B writers in one database transaction;
  - finalize counters, `completed` versus `partially_completed`, and `completed_at`;
  - prove rollback, retry, concurrency and account isolation in PostgreSQL.

## Required input boundary

Step 5G may post only an import row satisfying all of the following:

```text
status = pending
normalized_data is an object
normalized_data.deduplication = {schema_version: 1, status: unique}
normalized_data.posting_intent is a valid schema-version-1 successful intent
deduplication_key is present
created_transaction_id is null
created_investment_event_id is null
```

The stored posting intent is untrusted persistence data. Before any write, the posting
implementation must remove workflow metadata from a copy of `normalized_data`, run the
pure classifier again, serialize it with `model_dump(mode="json")`, and require exact
equality with the stored intent.

Rows in `needs_review`, `duplicate`, `skipped` or `failed` states are never posted.

## Canonical ownership

- `ImportBatch` and `ImportRow` remain owned by the imports module.
- `Transaction`, `InvestmentEvent` and `InvestmentMovement` are canonical history.
- Posting services may coordinate both domains, but must not move financial inference
  out of the existing pure classifier.
- Holdings, snapshots and read models remain downstream derived data.

## Global invariants

- One import row may create at most one canonical root entity.
- A transaction row sets only `created_transaction_id`.
- An investment row sets only `created_investment_event_id`.
- Created IDs and row status change in the same database transaction as the entity insert.
- An exact replay returns the existing entity and performs no semantic mutation.
- A stored ID whose entity is missing or differs from the expected canonical mapping is
  corruption and must fail closed.
- No posting path uses `float`; monetary and quantity values remain `Decimal`.
- Error messages and logs must not echo raw rows, descriptions, identifiers or amounts.
- No historical canonical entity is silently updated to reconcile a mismatch.

## Batch finalization contract

Owned by 5G-C:

```text
rows_total = all persisted rows
rows_imported = imported rows
rows_skipped = duplicate + skipped + failed + needs_review
completed_at = current UTC timestamp
```

Final status:

- `completed` when every persisted row is `imported`, `duplicate` or `skipped`;
- `partially_completed` when at least one row is imported and at least one row remains
  `failed` or `needs_review`;
- posting is rejected if no postable row exists and the batch has not already reached an
  exact valid terminal state.

The final response contract and exact zero-import policy are locked in 5G-C after both
row writers exist.

## Out of scope for all 5G substeps

- No holdings recalculation.
- No account or net-worth snapshots.
- No transfer pairing.
- No category or counterparty inference.
- No reporting-currency conversion.
- No automatic correction of historical entities.
- No background queue.
- No deletion or purge workflow.
- No schema migration unless a later substep proves an unavoidable database invariant
  that cannot be enforced safely with the inherited schema.

## Verification philosophy

Each substep must include focused unit tests and real PostgreSQL integration tests. Mock
sessions are not sufficient evidence for entity persistence, row linkage, rollback or
idempotent replay.

The final 5G-C PostgreSQL suite must exercise the complete persisted pipeline:

```text
register → parse → normalize → deduplicate → classify → post
```

## Completion

Step 5G is complete only after 5G-A, 5G-B and 5G-C are merged and mixed transaction and
investment batches post atomically with deterministic final state.
